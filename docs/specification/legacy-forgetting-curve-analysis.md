# Анализ кривой забывания и логики мастерства цветков

## 1. Обзор архитектуры

### 1.1 Формула забывания

Реализована классическая кривая Эббингауза: `R = e^(-t/S)`, где:

- **R** — retention (удержание), от 0 до 1
- **t** — время с последнего показа (в днях)
- **S** — стабильность памяти (в днях), зависящая от количества повторений

Стабильность вычисляется по формуле `S = S0 * multiplier^step`, где:
- `S0 = 0.9` дней (базовая стабильность нового материала)
- `multiplier = 2.2` (рост стабильности при каждом повторении)
- `step` — текущий индекс в лестнице интервалов

### 1.2 Лестница интервалов

Фиксированные интервалы повторения (в днях):
```
[1, 2, 4, 7, 10, 14, 20, 28, 42, 56]
```
Итого 10 шагов. Максимальный интервал — 56 дней (~8 недель).

### 1.3 Порог мастерства

- **MASTERY_THRESHOLD = 150** — количество уникальных показов карточек для 100% закрепления
- Формула: `masteryPercent = uniqueCardShows / 150`, ограничено сверху до 1.0
- Только режимы VOICE и KEYBOARD увеличивают mastery; WORD_BANK исключён

### 1.4 Состояния цветка и переходы

```
LOCKED -> SEED -> SPROUT -> BLOOM -> WILTING -> WILTED -> GONE
```

Определение состояния в `determineFlowerState()`:
1. Если `healthPercent <= 0.51` (WILTED_THRESHOLD + epsilon) -> **WILTED**
2. Если `healthPercent < 1.0` (но > 0.51) -> **WILTING**
3. Иначе — определяется по mastery:
   - `masteryPercent < 0.33` -> **SEED**
   - `masteryPercent < 0.66` -> **SPROUT**
   - `masteryPercent >= 0.66` -> **BLOOM**

Дополнительно:
- Если `mastery == null` или `uniqueCardShows == 0` -> **SEED**
- Если `daysSinceLastShow > 90` -> **GONE**

### 1.5 Здоровье цветка (healthPercent)

Вычисляется в `SpacedRepetitionConfig.calculateHealthPercent()`:
1. Если `daysSinceLastShow <= 0` -> 100%
2. Если `daysSinceLastShow >= 90` -> 0% (цветок исчез)
3. Если `daysSinceLastShow <= expectedInterval` -> 100% (вовремя)
4. Иначе: `health = 0.5 + 0.5 * e^(-overdueDays/stability)`, где `overdueDays = daysSinceLastShow - expectedInterval`

Здоровье никогда не падает ниже 0.5 (WILTED_THRESHOLD), кроме случая GONE (0%).

### 1.6 Масштаб цветка (scaleMultiplier)

`scale = max(0.5, masteryPercent * healthPercent)`

Произведение двух величин: прогресса освоения и текущего здоровья. Минимум 50%, максимум 100%.

### 1.7 Продвижение интервала

При каждом показе карточки (`MasteryStore.recordCardShow()`):
1. Вычисляется `daysSinceLastShow` с предыдущего показа
2. Проверяется `wasRepetitionOnTime(daysSinceLastShow, currentStep)`
3. Если вовремя — шаг продвигается вперёд: `nextIntervalStep(step, true) = step + 1`
4. Если с опозданием — шаг остаётся прежним: `nextIntervalStep(step, false) = step`
5. Шаг не может быть уменьшен — только вперёд или на месте

### 1.8 Поток данных

```
Пользователь отвечает правильно (VOICE/KEYBOARD)
  -> TrainingViewModel.recordCardShowForMastery(card)
    -> Проверка InputMode != WORD_BANK
    -> resolveCardLessonId(card) — определяет урок по ID карточки
    -> MasteryStore.recordCardShow(lessonId, languageId, cardId)
      -> Обновление uniqueCardShows (если карточка новая)
      -> Обновление shownCardIds (множество показанных карточек)
      -> Обновление lastShowDateMs
      -> Расчёт нового intervalStepIndex
      -> persistToFile() через AtomicFileWriter
  -> refreshFlowerStates()
    -> Для каждого урока: FlowerCalculator.calculate(mastery, totalCards)
    -> Обновление UI state
```

## 2. Найденные проблемы

### 2.1 Критичные

#### 2.1.1 `completedAtMs` сериализуется как 0L вместо null

**Файл:** `MasteryStore.kt`, строка 225

```kotlin
"completedAtMs" to (mastery.completedAtMs ?: 0L)
```

При сериализации `null` (урок не завершён) записывается как `0L`. При десериализации (строка 60):
```kotlin
completedAtMs = (lessonData["completedAtMs"] as? Number)?.toLong()
```
Значение `0L` будет прочитано как `0L` (не null), что означает, что **первая загрузка данных из файла пометит все начатые уроки как завершённые** с `completedAtMs = 0`.

**Влияние:** `markLessonCompleted()` проверяет `if (existing.completedAtMs != null) return` — после загрузки из файла урок навсегда считается завершённым, и повторная отметка невозможна. Это влияет на логику, которая зависит от статуса завершения урока.

**Статус:** КРИТИЧНО. Нарушает корректность определения завершённых уроков после перезагрузки приложения.

#### 2.1.2 Увеличение интервала не зависит от правильности ответа

**Файл:** `TrainingViewModel.kt`, строка 466; `MasteryStore.kt`, строка 123

`recordCardShowForMastery()` вызывается **только при правильном ответе** (внутри `if (accepted)` в `submitAnswer()`). Однако `recordCardShow()` в MasteryStore всегда продвигает интервал, если повторение «вовремя». Проблема в том, что продвижение интервала не учитывает, насколько уверенно пользователь ответил:

- Пользователь ответил правильно с 3-й попытки -> интервал всё равно продвигается
- Нет механизма отката интервала при слабом ответе

**Влияние:** Интервалы могут расти слишком быстро для карточек, которые пользователь фактически не знает. Цветок будет показывать BLOOM/SPROUT для плохо усвоенного материала.

**Статус:** СРЕДНЕЕ. Не ломает базовую логику, но снижает педагогическую ценность системы.

#### 2.1.3 `LessonLadderCalculator` добавляет +1 к дням (несогласованность)

**Файл:** `LessonLadderCalculator.kt`, строка 19

```kotlin
val daysSince = ((nowMs - mastery.lastShowDateMs) / DAY_MS).toInt().coerceAtLeast(0) + 1
```

В отличие от `FlowerCalculator.calculateDaysSince()` и `SpacedRepetitionConfig.calculateHealthPercent()`, которые вычисляют `daysSince` как целочисленное деление без +1, `LessonLadderCalculator` добавляет единицу. Это приводит к:

- Один и тот же урок может считаться «просроченным» в LessonLadder, но «здоровым» в FlowerCalculator
- Пользователь видит в лестнице «Просрочка+1» в тот же день, когда цветок ещё зелёный

**Влияние:** Несогласованность UI между лестницей уроков и визуализацией цветков. Пользователь может видеть конфликтующую информацию.

**Статус:** СРЕДНЕЕ. Визуальная несогласованность, может сбивать с толку.

### 2.2 Средние

#### 2.2.1 Нет отката интервала при длительном отсутствии

**Файл:** `SpacedRepetitionConfig.kt`, `nextIntervalStep()`

Когда пользователь возвращается после долгого перерыва (например, 30+ дней), `wasRepetitionOnTime()` возвращает `false`, и шаг остаётся прежним. Это корректно. Однако нет механизма **постепенного отката** — если пользователь пропустил 60 дней, его интервал остаётся на шаге 7 (28 дней), хотя память уже сильно деградировала.

**Рекомендация:** Добавить понижение шага пропорционально времени отсутствия. Например, если просрочено в 2+ раза, откатить на 1 шаг; если в 5+ раз — на 2 шага.

#### 2.2.2 `daysSinceLastShow = 0` при первом показе не продвигает интервал

**Файл:** `MasteryStore.kt`, строки 115-127

```kotlin
val daysSinceLastShow = if (existing?.lastShowDateMs != null && existing.lastShowDateMs > 0) {
    ((now - existing.lastShowDateMs) / (24 * 60 * 60 * 1000)).toInt()
} else {
    0
}
val newStep = if (existing != null && daysSinceLastShow > 0) {
    SpacedRepetitionConfig.nextIntervalStep(currentStep, wasOnTime)
} else {
    currentStep
}
```

Условие `daysSinceLastShow > 0` означает: если пользователь повторил карточку **в тот же день**, интервал не продвигается. Это корректное поведение (повторение в тот же день не считается повторением для интервала).

Однако есть нюанс: `daysSinceLastShow = 0` -> `wasOnTime` всё равно вычисляется как `true` (0 <= expectedInterval), но результат не используется из-за условия `daysSinceLastShow > 0`. Это не баг, но делает код запутанным.

**Статус:** МИНОРНОЕ. Код работает корректно, но может быть упрощён для читаемости.

#### 2.2.3 FlowerState не может «ожить» без нового показа

Цветок может перейти из BLOOM -> WILTING -> WILTED просто с течением времени. Чтобы вернуться из WILTING/WILTED в BLOOM, пользователь должен показать карточку (правильный ответ VOICE/KEYBOARD). После показа `lastShowDateMs` обновляется, `daysSinceLastShow` сбрасывается в 0, и здоровье возвращается к 100%.

Это **корректное поведение** по замыслу (повторение оживляет цветок), но стоит отметить, что цветок не может «частично» восстановиться — только полный сброс до 100% при новом показе.

**Статус:** НЕ БАГ, архитектурное решение.

#### 2.2.4 `recordCardShowForMastery` вызывается при перемещении назад

**Файл:** `TrainingViewModel.kt`, строка 723

```kotlin
fun prevCard() {
    ...
    prevCard?.let { recordCardShowForMastery(it) }
    ...
}
```

При нажатии кнопки «назад» (`prevCard()`) показ карточки записывается для mastery. Это означает, что пользователь может увеличить mastery, просто листая туда-сюда между карточками.

Однако стоит отметить, что `recordCardShow()` увеличивает `uniqueCardShows` только если карточка новая (не была показана ранее). А `shownCardIds` отслеживает показанные карточки. Таким образом, повторный показ одной и той же карточки **не увеличивает** `uniqueCardShows`, но **обновляет** `lastShowDateMs` и может продвинуть `intervalStepIndex`.

**Влияние:** Листание назад обновляет временную метку последнего показа, что искусственно «оживляет» цветок без реального обучения. Пользователь может поддерживать здоровье всех цветков, просто перелистывая карточки.

**Статус:** СРЕДНЕЕ. Потенциальное манипулирование системой повторения.

#### 2.2.5 `recordCardShowForMastery` вызывается в `startSession()` для текущей карточки

**Файл:** `TrainingViewModel.kt`, строка 191 и 1639

```kotlin
// init block:
_uiState.value.currentCard?.let { recordCardShowForMastery(it) }

// startSession():
currentCard()?.let { recordCardShowForMastery(it) }
```

При каждом запуске сессии (включая возобновление после паузы) показ текущей карточки записывается. Если пользователь ставит на паузу и возобновляет多次, один и тот же ответ может несколько раз обновлять `lastShowDateMs`.

Аналогично при `nextCard()` (строка 696): при переходе к следующей карточке, она сразу записывается как «показанная», даже если пользователь ещё не ответил на неё.

**Влияние:** Каждый запуск/возобновление сессии или переход к карточке обновляет `lastShowDateMs`, что может искусственно поддерживать здоровье цветка. Интервал может продвигаться, даже если ответ ещё не дан.

**Статус:** СРЕДНЕЕ. Преждевременная запись показа до ответа пользователя.

### 2.3 Минорные

#### 2.3.1 `totalCardsInLesson` передаётся, но не используется

**Файл:** `FlowerCalculator.kt`, строка 20

```kotlin
fun calculate(mastery: LessonMasteryState?, totalCardsInLesson: Int = 0): FlowerVisual
```

Параметр `totalCardsInLesson` принимается, но нигде не используется в методе. Мастерство всегда рассчитывается относительно `MASTERY_THRESHOLD = 150`, независимо от реального количества карточек в уроке.

**Статус:** МИНОРНОЕ. Мёртвый параметр, вводит в заблуждение.

#### 2.3.2 Нет синхронизации при доступе к MasteryStore из разных потоков

**Файл:** `MasteryStore.kt`

`MasteryStore` использует `MutableMap` для кэша без какой-либо синхронизации. В `TrainingViewModel` метод `recordCardShowForMastery()` вызывается из основного потока, а `refreshFlowerStates()` тоже из основного потока. Однако `viewModelScope.launch(Dispatchers.IO)` в блоке init вызывает `refreshFlowerStates()` через `withContext(Dispatchers.Main)`.

Если `forceReloadDefaultPacks()` в фоновом потоке завершится и вызовет `refreshFlowerStates()` в Main, а пользователь одновременно ответит на карточку, возможна конкурентная запись в кэш.

На практике Android Main thread — однопоточный, так что одновременная запись маловероятна, но при будущих рефакторингах это может стать проблемой.

**Статус:** МИНОРНОЕ. Текущая архитектура безопасна, но хрупка.

#### 2.3.3 `AtomicFileWriter` не гарантирует атомарность на некоторых Android FS

**Файл:** `AtomicFileWriter.kt`

Реализация использует паттерн «write-to-temp + rename», который не гарантирует атомарность на всех файловых системах Android. Метод `renameTo()` может вернуть `false` на некоторых устройствах.

Также есть `Thread.sleep(10)` в цикле удаления temp-файла — это Windows-специфичный workaround, который не нужен на Android. Поскольку это Android-приложение, код работает в эмуляторе/на устройстве с Linux FS, и этот workaround не нужен.

**Статус:** МИНОРНОЕ. Работает на практике, но `Thread.sleep` в файловых операциях — плохая практика.

#### 2.3.4 `calculateDaysSince` усекает, а не округляет

**Файл:** `FlowerCalculator.kt`, строка 99

```kotlin
return (diffMs / (24 * 60 * 60 * 1000)).toInt().coerceAtLeast(0)
```

Целочисленное деление означает, что если с последнего показа прошло 23 часа 59 минут, `daysSince = 0`. А если 24 часа 1 минута — `daysSince = 1`. Это корректное поведение для «дневных» интервалов, но стоит учитывать при тестировании.

Аналогичное поведение в `MasteryStore.recordCardShow()` (строка 116) и `SpacedRepetitionConfig.calculateHealthPercent()` (вызывается с int).

**Статус:** НЕ БАГ. Согласованное поведение.

#### 2.3.5 `done_threshold` не проверяет >= в `calculateHealthPercent`

**Файл:** `SpacedRepetitionConfig.kt`, строки 109-110

```kotlin
if (daysSinceLastShow >= GONE_THRESHOLD_DAYS) return 0f  // >= 90
```

Но в `FlowerCalculator.kt`, строка 40:

```kotlin
if (daysSinceLastShow > SpacedRepetitionConfig.GONE_THRESHOLD_DAYS) {  // > 90
```

`FlowerCalculator` проверяет `> 90`, а `calculateHealthPercent` проверяет `>= 90`. Это означает:
- При ровно 90 днях `calculateHealthPercent` вернёт 0.0
- При ровно 90 днях `FlowerCalculator` НЕ войдёт в блок GONE (т.к. `90 > 90` = false)
- Тогда `calculateHealthPercent` вернёт 0.0 для `daysSinceLastShow = 90`
- Но `FlowerCalculator` получит `healthPercent = 0.0`, и `determineFlowerState` вернёт WILTED (т.к. 0.0 <= 0.51)

Однако в тесте `calculate_exactly90Days_notGoneYet` ожидается WILTED (не GONE). Тест проходит потому, что `FlowerCalculator` использует `>`, а не `>=`. Но `calculateHealthPercent` для 90 дней вернёт 0.0, что может быть неожиданным.

Состояние при 90 днях: **WILTED** (не GONE). При 91 дне: **GONE**. Тесты это подтверждают.

**Статус:** МИНОРНОЕ. Несогласованность порогов между двумя методами, но работает корректно благодаря порядку проверок в `FlowerCalculator`.

## 4. Потенциальные улучшения

### 4.1 Гибкая формула мастерства

Текущая формула `uniqueCardShows / 150` линейна. Можно ввести:

- **Взвешенное мастерство**: Первые показы дают больше прогресса, последние — меньше (логарифмическая кривая)
- **Зависимость от урока**: `MASTERY_THRESHOLD` фиксирован на 150, но уроки могут содержать разное количество карточек. Для урока с 50 карточками 150 показов — это 3 полных цикла, а для урока с 300 карточками — только половина первого прохода.

### 4.2 «Мягкое» увядание

Сейчас здоровье сбрасывается до 100% при любом показе. Можно ввести частичное восстановление: если пользователь ответил правильно, здоровье восстанавливается до 80-90%, а не до 100%. Для полного восстановления — несколько последовательных правильных ответов.

### 4.3 Индивидуальные интервалы для карточек

Сейчас `intervalStepIndex` привязан к **уроку**, а не к карточке. Все карточки в уроке делят один и тот же интервал, даже если пользователь хорошо знает одни и плохо — другие. В перспективе можно хранить интервалы для каждой карточки индивидуально (как в SuperMemo/Anki).

### 4.4 Визуальная обратная связь при увядании

Цветок переходит из BLOOM -> WILTING -> WILTED без промежуточных уведомлений. Можно добавить push-уведомление при переходе в WILTING: «Ваш цветок по уроку X начал увядать! Повторите урок.»

### 4.5 Статистика кривой забывания

Добавить аналитику: отслеживать, как часто пользователи повторяют вовремя vs с опозданием. Это поможет настроить параметры `BASE_STABILITY_DAYS` и `STABILITY_MULTIPLIER` на основе реальных данных.

### 4.6 Транзакционность операций mastery

Сейчас `recordCardShow()` делает: (1) чтение кэша, (2) модификация, (3) запись в файл. Если приложение упадёт между (2) и (3), кэш будет изменён, но файл — нет. После перезапуска кэш перечитается из файла, и изменение потеряется. Стоит записывать в файл первым делом, а потом обновлять кэш.
